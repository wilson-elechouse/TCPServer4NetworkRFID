# TCPServer4NetworkRFID

Cloud TCP Broker + browser test page for Network RFID/NFC reader demos.

Primary firmware target: `ESP32S3_LF_HF_Network_RFID` in `ELECHOUSE Test` mode.

Primary hardware target: ELECHOUSE Network RFID Reader Mainboard with ESP32-S3 SuperMini controller and ST25R3916B HF frontend.

This project lets a browser test page and a hardware RFID reader communicate through a small cloud broker:

- The browser opens a test page and receives a temporary session code.
- The RFID device connects to one fixed TCP endpoint as a plain TCP client.
- The first packet from the device is `HELLO <SESSION_CODE> <DEVICE_ID>\n`.
- The broker binds that device TCP socket to the browser session.
- LF/HF card data uploaded by the device appears in the browser in real time.
- The browser can also send firmware commands back to the device through the same TCP socket.
- A separate no-login serial configuration page can send the same firmware commands through browser Web Serial.

The device firmware does **not** need HTTP, HTTPS, or WebSocket support. It only needs a normal TCP client. The current ELECHOUSE firmware uploads card events as JSON Lines in `ELECHOUSE Test` mode.

## Features

- Plain TCP device endpoint.
- Browser UI served by the same Node.js service.
- Browser ↔ broker channel uses WebSocket.
- Random 8-character session code.
- Session restore on browser refresh using a broker session cookie.
- **New session** closes/replaces the previous live session for that browser/user.
- **End session** immediately closes the web socket, disconnects the device socket, deletes the in-memory session, and clears the browser session cookie.
- Session auto-expiry cleanup.
- Device `PING\n` → broker `PONG\n` heartbeat support.
- Optional text/hex display. Hex is generated by the broker from received bytes and is hidden by default.
- Basic per-IP TCP connection and auth-failure limits.
- Optional WordPress-login gate for the browser UI/API. This is **disabled by default** in this repository.
- No-login Web Serial configuration tool at `/serial` for USB CDC/product UART setup.

## Architecture

```text
Browser test page
  | HTTPS page + WebSocket
  v
Node.js Broker
  | plain TCP
  v
RFID/NFC device firmware
```

Default production-style endpoints:

```text
Browser page: https://www.example.com/rfid-tcp-broker/
Serial page:  https://www.example.com/rfid-tcp-broker/serial
WebSocket:    wss://www.example.com/rfid-tcp-broker/ws?code=<CODE>
Device TCP:   www.example.com:9000
First packet: HELLO <CODE>\n
Local HTTP service: 127.0.0.1:19090
Local TCP service:  0.0.0.0:9000
```

## Quick start: local development

```bash
git clone https://github.com/wilson-elechouse/TCPServer4NetworkRFID.git
cd TCPServer4NetworkRFID
npm install
HTTP_PORT=19190 \
TCP_PORT=19191 \
PUBLIC_TCP_HOST=127.0.0.1 \
PUBLIC_BASE_PATH=/rfid-tcp-broker \
node server.js
```

Open the local page directly:

```text
http://127.0.0.1:19190/
http://127.0.0.1:19190/serial
```

Run a local end-to-end smoke test in another terminal:

```bash
API=http://127.0.0.1:19190 \
DEVICE_HOST=127.0.0.1 \
DEVICE_PORT=19191 \
node scripts/smoke-test.js
```

Expected result ends with:

```text
SMOKE_OK
```

## Device protocol

Device connects to the TCP port, then sends one line within 10 seconds:

```text
HELLO <SESSION_CODE> <DEVICE_ID>\n
```

Example from the current firmware:

```text
HELLO A7K3Q9M2 ELECHOUSE_RFID-68:EE:8F:DD:ED:04\n
```

Successful reply:

```text
OK A7K3Q9M2\n
```

Then the current firmware uploads card events as JSON Lines:

```text
{"type":"card","band":"HF","card_type":"ISO14443A","uid":"1C 2C FB 10","ms":1002132}\n
```

The broker also accepts plain text lines for simple demos, but the current `ESP32S3_LF_HF_Network_RFID` firmware uses the JSON Lines format above in `ELECHOUSE Test` mode.

See [`docs/FIRMWARE_INTEGRATION.md`](docs/FIRMWARE_INTEGRATION.md) for firmware details, configuration commands, and troubleshooting.

## Configuration

Configuration is controlled by environment variables.

| Variable | Default | Description |
| --- | --- | --- |
| `HTTP_HOST` | `127.0.0.1` | HTTP/WebSocket bind address |
| `HTTP_PORT` | `19090` | HTTP/WebSocket port |
| `TCP_HOST` | `0.0.0.0` | Device TCP bind address |
| `TCP_PORT` | `9000` | Device TCP port |
| `PUBLIC_TCP_HOST` | `www.elechouse.com` | Host shown to device users |
| `PUBLIC_BASE_PATH` | `/rfid-tcp-broker` | Public reverse-proxy path |
| `SESSION_TTL_MS` | `1800000` | Session lifetime, default 30 minutes |
| `SESSION_CODE_LENGTH` | `8` | Random session code length |
| `MAX_SESSIONS` | `200` | Max in-memory sessions |
| `MAX_WEB_CLIENTS_PER_SESSION` | `8` | Browser viewers per session |
| `HELLO_TIMEOUT_MS` | `10000` | Time allowed for device `HELLO` |
| `MAX_HELLO_BYTES` | `512` | Max first-packet bytes |
| `MAX_DEVICE_CHUNK_BYTES` | `65536` | Max device chunk bytes |
| `MAX_WS_PAYLOAD_BYTES` | `65536` | Max browser command payload bytes |
| `MAX_TCP_CONNECTIONS_PER_IP` | `5` | TCP concurrent connection limit per IP |
| `MAX_TCP_AUTH_FAILURES_PER_IP` | `300` | Invalid `HELLO` limit before temporary block |
| `TCP_AUTH_FAILURE_WINDOW_MS` | `600000` | Failure count window |
| `TCP_AUTH_BLOCK_MS` | `600000` | Temporary block duration |
| `REQUIRE_WEB_AUTH` | `0` | Optional login gate. `0`/unset = disabled, `1` = enabled |
| `WORDPRESS_AUTH_CHECK_URL` | `http://127.0.0.1/wp-json/elechouse-rfid/v1/auth-check` | Local WordPress auth endpoint when auth is enabled |
| `WORDPRESS_AUTH_HOST` | `www.elechouse.com` | Host header used for local WordPress auth check |
| `AUTH_CACHE_MS` | `15000` | WordPress auth-check cache time |
| `SESSION_COOKIE_NAME` | `elechouse_rfid_session_code` | Browser cookie name for session restore |

## Optional WordPress login gate

The login gate code is included but **disabled by default**.

Default behavior:

```text
REQUIRE_WEB_AUTH unset or 0 → no WordPress login required
```

Enable it only when deploying behind a WordPress site:

1. Install the MU plugin:

   ```bash
   mkdir -p /path/to/wordpress/wp-content/mu-plugins
   cp wordpress/elechouse-rfid-broker-auth.php \
      /path/to/wordpress/wp-content/mu-plugins/elechouse-rfid-broker-auth.php
   ```

2. Set broker environment variables:

   ```bash
   REQUIRE_WEB_AUTH=1
   WORDPRESS_AUTH_CHECK_URL=http://127.0.0.1/wp-json/elechouse-rfid/v1/auth-check
   WORDPRESS_AUTH_HOST=www.example.com
   ```

3. The WordPress endpoint is local-only. External requests return `403 local_only`.

When enabled:

- Browser page `/` requires a valid WordPress logged-in cookie.
- `/api/session`, `/api/session/close`, and `/api/stats` require login.
- `/api/stats` requires an administrator role.
- Serial configuration page `/serial` does **not** require login. It only uses the browser Web Serial API to access a user-selected local serial port and does not create broker sessions.
- Device TCP port remains open, but it only accepts valid active session codes.

## Production deployment example

### 1. Install files

```bash
sudo mkdir -p /opt/elechouse-rfid-tcp-broker
sudo rsync -az ./ /opt/elechouse-rfid-tcp-broker/
cd /opt/elechouse-rfid-tcp-broker
npm install --omit=dev
```

### 2. Start with PM2

Edit `ecosystem.config.cjs` for your domain/ports, then:

```bash
pm2 start ecosystem.config.cjs
pm2 save
```

### 3. Nginx reverse proxy example

```nginx
location /rfid-tcp-broker/ {
    proxy_pass http://127.0.0.1:19090/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
}
```

### 4. Firewall

Allow the device TCP port:

```bash
sudo ufw allow 9000/tcp comment 'RFID TCP broker'
```

## Session cleanup and data retention

Sessions are in-memory only. The broker does **not** create per-user or per-session files.

Cleanup paths:

- **End session** button/API: immediate cleanup.
- **New session**: closes/replaces the previous session for that browser/user.
- **TTL expiry**: cleanup timer removes expired sessions every 30 seconds.
- **Process restart**: all in-memory sessions disappear and sockets close.

`expireSession()` closes device sockets, closes web sockets, broadcasts the close reason, and removes the session from the in-memory `Map`.

Normal PM2/stdout logs may remain for debugging. They are service logs, not per-user session files.

## Security notes

- Keep the browser page behind HTTPS when exposed publicly.
- Keep `REQUIRE_WEB_AUTH=0` for standalone demos, or set `REQUIRE_WEB_AUTH=1` behind WordPress if only logged-in users should create sessions.
- TCP `9000` must remain reachable by devices, but invalid/expired session codes are rejected.
- Increase/decrease `MAX_TCP_AUTH_FAILURES_PER_IP` depending on device reconnect behavior.
- For public production use, monitor logs and consider firewall/WAF rules for abusive IPs.
